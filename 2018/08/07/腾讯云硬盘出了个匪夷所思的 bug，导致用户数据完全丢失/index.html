腾讯云硬盘出了个匪夷所思的 bug，导致用户数据完全丢失

今日热点 知乎日报 2018/08/07

简而言之：匪夷所思。
在腾讯云硬盘的官网上多次提到了多机备份，以下为文档原话：

在每个存储写入请求返回给用户之前，CBS 就已确保数据已被成功写入三份，且跨机架存储
其教学视频也明确指出了多机备份属性。
事后腾讯解释数据丢失是因为一个罕见的硬件固件 bug。硬件问题本身并没有什么特别的，这在大的数据中心是常态而非异常。但问题是三台机器硬件同时触发了 bug 造成数据全部丢失。
同一个硬件 bug 搞丢了三台机子上的数据，这可能是 2018 年最锲而不舍的 bug 了。
如果腾讯云本身没有宣传有多机备份，那么这家创业公司自己不做备份完全是自己的问题。现在腾讯云在官网上反复宣传强调了多机备份，企业自己不做备份也可以理解(尤其这家公司并不是软件互联网公司)，毕竟在绝大多数场景中三份备份已经完全足够。
因为多机备份是分布式系统高可用性保证的重要一环，所以腾讯除了对这家创业公司需要作出解释外，更需要向其所有云用户解释为何其多机备份会同时受到影响。
关于公司使用的是否是云硬盘：
我写这份答案的时候参考的是云头条发布的文章：
腾讯云硬盘故障，导致用户「数据完全丢失」
文内多次提到了是云硬盘的问题，所以我在回答中也都是根据云硬盘的资料。评论中有同学提到这家公司用的是 CVM 本地磁盘没有多机容灾，从这篇文章中我找不到相关的内容。
关于副本和备份：
有很多同学也提到了副本和备份的问题，认为腾讯提供的是多副本，公司还得自己做备份。
我不是专业的 DBA，所以意见不够专业。
不过我认为，在有副本的情况下还需要做备份的看法是完全正确的，但是这和现在的问题没有太大关系。
副本的作用是将一份数据拷贝到多台机器，它能够应对的场景有：

  1. 机器故障 。如果是非同步复制的话，可能会有部分数据丢失。但丢失的数据量不会太大，主要取决于网络延迟。
  2. 自动恢复 (在一些系统中当副本数低于一个临界点时，系统会将副本的数据多拷贝几份到其他机器以防止数据完全丢失。当机器恢复时再清理多余的副本)
    它不能应对的场景有：
  3. 操作错误。比如某 DBA 完成了一波从删库到跑路的操作，那么所有副本的数据都会被删，这种情况下副本没有办法直接恢复数据。
  4. 代码错误。一般副本都是复制最后写入的数据，如果因为代码错误数据在写入前就已经出错，那么副本也无法恢复数据。比如，在一些系统中，一个大的文件会被分割成多个小文件再被写入，如果分割的逻辑有问题，那么所有副本同时都会出错。
    备份是将现有的数据存储起来，它能应对的场景有：
  5. 机器故障。但是需要手动恢复备份，其丢失的数据比起副本会更多些。丢失的数据量取决于备份周期，从一天到数天不等。
  6. 操作 / 代码错误。(就算 DBA 删库也不慌)
    它不能应对的场景有：
  7. 自动恢复。一般情况下副本需要在检测到问题后，由运维手动恢复。
    所以，如果那家公司有备份，那么数据丢失的严重性不会有那么严重，而且公司也应该要防备各种人为或软件的错误。所以我赞同有副本的情况下也得做备份。
    但是，腾讯给出的解释是硬盘固件 bug。这种情况副本应该足够恢复数据，所以我不认为备份和本次事故有太大关系。
